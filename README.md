# Sales Dashboard Agent

[![CI](https://github.com/saiabhi26/agent-chat-for-sales/actions/workflows/ci.yml/badge.svg)](https://github.com/saiabhi26/agent-chat-for-sales/actions/workflows/ci.yml)

A real-time sales analytics dashboard with a chat agent that turns plain English into filters — and **remembers when you correct it**.

### ▶ Live: **https://agent-chat-for-sales.vercel.app**

> ⚠️ **First load takes 20–30 seconds. Please be patient.**
> The backend runs on a free tier that spins down after 15 minutes of inactivity, so the first request has to wake the container. The page itself paints instantly; only the data waits. Every request after that is fast.

---

## What it does

- **Analytics that update live.** Revenue, deal count, average deal size, top rep, top region. No refresh — new transactions are pushed to every connected browser over Server-Sent Events.
- **A chat agent over the data.** "Show me enterprise deals in the West over $50k" becomes a structured filter, not a SQL string. The agent reports its own confidence: clear queries run, ambiguous ones ask first.
- **Correction memory.** Tell it "John Smith means Mike Chen" once, and it applies that on *every future phrasing* — including abbreviations. This is stored, scoped to a field, and survives the session.
- **Drift detection.** When a new deal moves a region's average by ≥10%, the agent proactively says so, unprompted.
- **Spend guardrails.** The agent costs real money per query, so it is wrapped in a rate limiter and a daily budget (below).

## Architecture

```
Browser ──────────────► Vercel (Next.js, CDN)
   │                          │
   │  SSE + fetch             │
   ▼                          ▼
Render (Docker, Hono) ──► SQLite (seeded from a committed fixture)
   │
   └──► Anthropic API  ── behind rate limit + daily budget
```

**The frontend and backend are hosted separately, on purpose.** The backend's free tier sleeps when idle. If it also served the HTML, a cold start would mean staring at a blank browser tab for half a minute. Splitting them means Vercel's CDN paints the full UI immediately and only the *data* is pending — a loading state instead of a void.

Inside the backend, the layering is strict:

- `db/queries.ts` — the only file that touches SQLite.
- `routes/` — HTTP only. No business logic.
- `services/` — all logic, one job per file, each independently importable and independently tested.

The database is seeded from a **committed `seed.json`**, validated against a schema by a test. It is deliberately *not* generated by an LLM at boot: that made startup slow, non-deterministic, and impossible to write assertions against.

## Engineering techniques

The parts I'd actually want to talk through in an interview.

**Test-driven, and the tests cost $0 to run.**
Every module was written test-first — red, green, refactor. The interesting constraint was that the two riskiest subsystems (spend guardrails and time-based budgets) are the ones you least want to test against a live API or a real clock. So **the Anthropic client and the clock are both injected**. The whole 67-test suite runs in ~400ms with no network and no spend. The deployed app is where I *confirm* behavior, not where I test it.

**Layered spend defense.** The agent has a live API key in production, which is only safe because of three independent layers:

| Layer | Stops | Survives |
|---|---|---|
| 5 queries / IP / minute (sliding window) | one loud caller | — |
| 100 calls / UTC day (global) | the aggregate bill, including distributed floods | not a container restart |
| Hard $5 account balance, auto-reload **off** | everything, absolutely | my code being wrong |

Per-IP limiting alone is not enough, and that's the point worth noticing: 1,000 IPs each sending 5/min are all individually well-behaved, and the bill is still enormous. One API key means one global budget. The bottom layer exists because **code cannot defend its own budget across a process restart** — a truly hard cap has to live outside the process.

**Trusting `X-Forwarded-For` correctly.** A client can forge that header; the proxy *appends* the address it actually saw. So the real IP is the **last** entry, not the first. Taking the first would let anyone bypass the rate limiter by rotating a fake header — a limiter that is present but not actually limiting, which is worse than none, because you'd trust it.

**CI is the only route to production.** Render's auto-deploy is disabled. A push to `main` runs typecheck → tests → build, and only then triggers the deploy — so the pipeline actually gates something instead of racing it. The deploy step then **polls until the live health endpoint reports the new commit SHA**, because the old container keeps serving while the new one builds and a naive 200-check would happily pass against the previous code. It then smoke-tests production: the fixture seeded, the agent is configured, the guardrails are in the running container, and SSE really streams.

**Correction memory that actually fires.** The first version keyed corrections on the raw user sentence, so the memory only re-triggered on a *verbatim* repeat — it had never once fired for a real user. Rewritten to key on the **parsed filter value, scoped to a field**, so a correction on "John Smith" → `salesRep` fires on any phrasing that resolves to John Smith. Matching is whole-value and case-insensitive, and nothing is compiled as a regex — stored terms are user input, and `new RegExp(userInput)` is both a crash (a stored `[` throws) and a ReDoS vector.

## Known limitations

Honest list; these are choices, not oversights.

- **The agent's system prompt is injectable.** A determined user can talk it into ignoring its instructions. Real mitigation is output validation on the parsed filters, which is where I'd spend the next hour.
- **`computeAnalytics` full-table-scans on every insert.** Fine at 120 rows, wrong at 120,000. It should be an incremental aggregate.
- **Corrections are global and have no undo.** On a public link, someone will eventually teach it nonsense, and it will persist until the container restarts.
- **Guardrail state is in-memory**, so it resets on container restart. That's why the account-level spend cap exists.
- **No FX conversion** — amounts are treated at their stated value.
